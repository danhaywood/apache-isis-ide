3. Naked Objects Application Library

As described in the programming model, the application library primarily consists of a number of annotations that refine the semantics of the domain object model. These reside in the org.nakedobjects.applib.annotation package. These are supplemented by javax.persistence.Entity to represent a domain (entity) object.

The application library has a number of other packages:

  • The org.nakedobjects.applib package consists primarily of two abstract classes AbstractDomainObject and AbstractFactoryAndRepository. These are convenience (non-mandatory!) superclasses for domain classes and services respectively. In addition it defines the DomainObjectContainer interface, discussed later.

  • The org.nakedobjects.applib.security package contains classes that represent the logged-on user's identity. These can be returned by the Domain Object Container.

  • The org.nakedobjects.applib.util package has a couple of helper classes useful for constructing return values for the title() method or the validateXxx and disableXxx methods.

  • The org.nakedobjects.applib.value package contains a number of classes that can be used instead of Java's own primitives and wrapper classes for properties. The prime benefit for using these value classes is (in some cases) more sophisticated rendering in the NOF UI, at the cost of arguably greater coupling to the NOA.

Over time we expect the application library to be extended as more sophisticated semantics are defined and supported by NOA implementations. At the same time some annotations may be deprecated if they can be replaced by JEE or other de-facto annotations if they represent the same semantics as required by NOA.
